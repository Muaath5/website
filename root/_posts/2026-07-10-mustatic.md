## Overview
This architecture replaces the traditional monolithic backend with a pre-compiled, serverless pipeline. It is built on three pillars:

- Data: JSON/Sqlite/CSV (Acts as your source of truth instead of a runtime SQL database)
- Builder: Python/Swift/Go (A script that handles heavy algorithmic calculations before page load)
- Engine: Jekyll/Liquid (High-speed aesthetic wrapper to compile Markdown/Frontmatter the final HTML, a bit redundent but useful)

## Structure
Here is the best structure to be followed:
- `.github/workflows/pages-deploy.yml`
- `builder/`
  - `lib/`
  - `build.py`
- `data/`
  - `*.json`
  - `*.csv`
- `root/`
  - `_layouts/`
  - `*.html`
  - `*.md`
  
  This seperates the core data, builder, and raw Markdown/HTML files that exists by default without building (like `404.html`).

Although the examples use this structure, which is not the best:
- `.github/workflows/`
- `root/`
  - `data/`
  - `_layouts/`
- `build.py`

## Deployment
The lifecycle of an update requires zero manual server administration:
1. Commit to edit data: A maintainer pushes updated JSON or CSV files directly to the Github repository
2. Github deployment workflow: Github Actions detects the push and provisions an isolated runner
3. Running Builder script: The runner executes python `build.py` (or in any language). This script synthesizes the data and injects it into the `./root` directory by writing files with their front-matter YAML format.
4. Engine hosting: The workflow executes `bundle exec jekyll serve -s ./root`, mapping the pre-calculated data into semantic HTML layouts and pushing the inert assets to GitHub Pages.

This architecture is designed to perfectly fit Github pages, but you can use Cloudflare pages or other services.

### Public Data API
Because your raw data is JSON/CSV, allow users to access it in `website.com/data/file.json` endpoints.

### Data Error checking in Builder
The Builder should first check the data format, read it and ensure it's valid, after that the building process comes in.
### Balance between Workflow and Javascript
Don't make fetching all the data comes in Javascript this will make it slow for the end user, you must benifit from the precomputing.

The thumb-rule says: Max of 3 seconds loading in Javascript, and Max of 3-5 minutes of workflow building, notice that the base Workflow with an empty `build.py ` takes around 20-30 seconds.

## Addons
### Secrets Management: Injecting keys via GitHub Secrets
### Localization (i18n): Using translation folders/JSON and Javascript routing.
Dual-language support (like AR/EN) can be achieved seamlessly by centralizing your strings in a `translations.json` file. The builder script duplicates routing paths, and localized strings are mapped via the engine's data folders.

### Encrypted Pages: The cryptography of hosting AES-encrypted endpoints
For sensitive content, the builder can generate an AES-encrypted static endpoint (e.g., `page.enc.html`). This is paired with a simple `view.html` helper interface, allowing users to enter a password and decrypt the DOM entirely on the client side without needing a backend server.

### Submodules (Proceed with Caution)
You can use Git Submodules (`.gitmodules`) to share core builders, layouts, and styles across multiple separate website repositories. While this adheres to DRY principles, debugging is notably difficult; submodule bugs often won't be discovered until a migration commit is tested.

## Pros & Cons
### Pros
- Serverless: You can code projects with budget absolute 0$
- Fast Edit: A single commit writing some JSON in your mobile would change the whole website in minutes
- Static + Database: You could have an easy access to website's data even if they are sensitive
- Traffic Analysis: You can link it with Google Analytics and achieve statistics without having a database
- Version Control & Rollbacks: Because your database state is tracked immutably via Git commits, every single change creates an audit trail. If a maintainer makes a mistake, reverting the website to a previous state is instantaneous.
- Bulletproof Security: Eliminating the runtime database removes the risk of SQL injections entirely, while dropping the backend server logic drastically minimizes exposure to XSS and DDoS attacks.
- Custom domain: you can use a custom domain that you rent from Cloudflare and even use an Email routing system, this gives you professional looking website

### Cons
- Readonly database: you cannot write to a database 
- Manual initiating: currently it doesnt have many documentations so most of the work must be done manually
- Build-Time Scaling Limits: As your dataset grows exponentially over the years, the builder script will take longer to compute. You are fundamentally bound by the time limits and processing power of the GitHub Actions runner environment.
- No Real-Time Capabilities: Implementing live features (such as WebSockets, live chat, or instantaneous real-time leaderboards) is completely impossible within the core architecture without hooking into a third-party external API.
